Database APIs

Monitoring data will be stored in a SQLlite DB. This module exposes all necessary APIs for the same.

The process monitor is detecting events with timestamps.

The next logical step is to persist these events rather than just printing them. That means building the storage module (db.py) with SQLite. 2 options in my mind 1. store as events - pid, app, start/ stop, timestamp (only inserts) 2. create new record at open - pid, app, start time, update stop time on stop based on pid and not sttopped (based on NULL stop time)

Option 1 (event log) — Simpler to write (inserts only), but querying “how long was Chrome open?” requires matching start/stop pairs afterwards. Option 2 (session record) — Slightly more complex (insert + update), but gives you duration directly. The risk: if the app or machine crashes, you’ll have orphan rows with NULL stop times.

Option 1 has less locking risk — it’s insert-only, so each write is a quick, independent transaction. Option 2 requires a SELECT to find the matching row, then an UPDATE, which holds the lock longer.

With Option 1 you can pair start/stop events in a SQL query when generating reports (e.g., using a self-join on PID). Want to start building the events table?

how do I write a query to so that start and stop event for a combination of pid and app can be extracted as a single row with duration calculated?

🤖Reply🤖

🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠

You can do this with a self-join — join the events table to itself, matching start events with stop events on the same pid and app. SQLite has a julianday() function that’s useful for calculating time differences.

The general shape is:

SELECT s.pid, s.app, s.timestamp AS start_time, e.timestamp AS stop_time
FROM events s JOIN events e ON s.pid = e.pid AND s.app = e.app
WHERE s.event_type = 'start' AND e.event_type = 'stop'

You can run raw SQL in fastlite with db.execute(sql).fetchall(). Try this query first, then we can add the duration calculation using julianday().

🤖Reply🤖

🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠🧠

The duration came out as 0 because SQLite is subtracting two text strings. You need to convert them to numeric values first using julianday():

(julianday(e.timestamp) - julianday(s.timestamp)) * 86400 AS duration_seconds

julianday() returns days, so multiplying by 86400 (seconds in a day) gives you duration in seconds. Try replacing your duration line with that.
